### 1.3 技术债与风险点

| 风险 | 说明 | 影响 |
|------|------|------|
| **无版本控制** | 项目非 Git 仓库 | 无法回溯变更、分支开发、协作 |
| **代码全在 ui.js** | 1220 行单文件，画廊/场地独立页会进一步膨胀 | 维护成本持续上升 |
| **无前端构建** | 原生 HTML/CSS/JS，依赖 CDN 加载 Chart.js 和 Supabase | 部署简单但离线不可用 |
| **Supabase 匿名直连** | anonKey 暴露在前端，未开 RLS | 数据安全风险（虽当前无敏感数据） |
| **支持注释缺失** | 代码几无注释 | 新人接手成本高 |
| **无错误边界** | UI 层没有 try/catch 兜底 | 单点异常可能白屏 |

## 二、P0 开发计划

P0 共 **7 项**，总预估工作量 **15-18 天**（单人开发）。

### 2.1 方案与策略

在执行 P0 前，建议做一次**代码重构铺垫**：

| 重构事项 | 原因 | 做法 |
|----------|------|------|
| **收入分类标准化** | 目前 gallery=otherAmount, venue=venueAmount，POS 已用 ticketItems/coffeeItems | 统一为 `revenue_items JSONB[]`，一个数组收所有类型 |
| **ui.js 拆分趋势** | POS 收银 + 画廊 + 场地 → 3 个独立功能区 | 收入相关渲染抽到 `ui-pos.js`、`ui-gallery.js`、`ui-venue.js` |
| **初始化 Git** | 后续迭代必须版本控制 | `git init` + `.gitignore` |

> `ui.js` 不一定要在 P0 前全部拆完，可以在每新增一个标签页时顺带拆分。

> 简单方案：不走 Supabase Auth，自建 users 表。密码用 `crypto.subtle.digest('SHA-256', ...)` 前端加密后比对。这样可以避免 Supabase Auth 的复杂配置，且数据仍在自己控制的表里。

## 三、执行建议

### 3.1 推荐执行顺序

```
迭代 1 画廊 → 迭代 2 场地 → 迭代 3 退款
→ 迭代 4 登录 → 迭代 5 日志
→ 迭代 6 对比分析 → 迭代 7 部署
```

理由：
- **画廊和场地**是独立的业务模块，与现有 POS 互不干扰，风险低
- **退款**改动只在 POS 模块内，但涉及数据模型变更，应趁早
- **登录+日志**是基础设施层，建议在两三天内连续完成，一次把所有 CRUD 的日志埋点做完
- **对比分析**纯粹是图表层增补，可以穿插在其他迭代排队时做
- **部署**最后做，系统验证齐全后再上线

### 3.2 一个迭代完成的标准

- [ ] 功能页面可渲染、可操作
- [ ] 数据可正常写入/读取 Supabase
- [ ] 编辑和删除功能正常
- [ ] 操作已自动记录操作日志（如果登录模块已完成）
- [ ] 无 console 报错

### 3.3 建议的工作方式

每启动一个迭代前，我会先确认具体需求细节，再进入开发。
